<p>geohot published "The Eternal Sloptember" a few weeks ago. The argument: AI agents frontload progress but never achieve polish. Low performers using agents produce high-volume output with no feedback loop to catch errors. Organizations with weak review culture drown in it. The post is about code quality, and the discourse around it has mostly stayed in that lane.</p>

<p>The SBOM provenance problem — documented <a href="/14-sbom-ai-provenance/">here last week</a> — is about audit trails. SBOM tooling accurately enumerates dependencies but records nothing about AI-generated code provenance: which model, which version, whether a human reviewed the output before it hit a protected branch.</p>

<p>These look like different problems. They are the same bug.</p>

<h2 id="the-failure-mode">The failure mode is different in security contexts</h2>

<p>A generated function that doesn't handle edge cases correctly will surface eventually. Tests fail. Users report issues. The feedback loop exists, even if it's slow.</p>

<p>A generated security configuration that's misconfigured doesn't always surface. An overly permissive IAM policy might never be exploited. A missing encryption-at-rest flag might not matter until it does. A firewall rule that opens the wrong port on the wrong network can sit there for months.</p>

<p>The failure mode for security slop isn't "it breaks." It's "it works fine right up until the blast radius is large."</p>

<p>I ran the geohot experiment concretely in May 2023: same infrastructure build, LLM-generated Terraform versus cookiecutter atomic modules with OPA/Rego policy gates, streamed live. The LLM introduced a VPC peering misconfiguration that would have connected the honeypot network to the canary network. Silently. <code>terraform plan</code> exited clean. <code>terraform validate</code> exited clean. The misconfiguration was only visible when I read the plan output and knew what the intended architecture was.</p>

<p>That's the failure mode: confident, syntactically valid, semantically wrong. The model optimized for "looks correct" because that's what its training data rewarded. It had no concept of what the architecture was supposed to accomplish.</p>

<h2 id="where-they-meet">Where the quality problem and the audit problem meet</h2>

<p>geohot's framing is that agents are force multipliers for high performers and slop generators for low performers. The differentiator is whether the person using the tool can evaluate the output.</p>

<p>The audit trail problem adds a second dimension: even when a senior engineer reviews AI-generated code and catches the quality issues, the provenance gap means the review isn't recorded. Six months later, during an incident, the timeline asks "who reviewed this and when?" The answer is "a senior engineer, verbally, in a PR comment that didn't mention the code was AI-generated." That's not a documentation trail. It's a memory.</p>

<p>The two problems compound. You can't audit what you can't trace, and what you can't trace was written by something that didn't understand the system.</p>

<p>When a senior engineer writes a security control they make implicit decisions based on context accumulated over weeks or months on that codebase. When a model writes the same control it makes statistically likely decisions based on what similar controls look like in its training data. Those are not the same thing, and the difference isn't visible in the output until something goes wrong.</p>

<h2 id="the-specific-risk">The specific risk for CI/CD pipelines</h2>

<p>The most dangerous version of this isn't a misconfigured IAM policy. It's AI-generated CI/CD pipeline code.</p>

<p>Pipelines are security infrastructure. They gate what gets deployed, who can trigger deployments, what secrets are accessible during builds, and what runs in production. A pipeline generated by a model that made statistically likely decisions about GitHub Actions syntax is a pipeline whose security properties weren't specified — they were guessed at.</p>

<p>The same model that confidently generates a VPC peering block will confidently generate a workflow that exposes secrets to pull request builds from forks, or that uses a floating action reference instead of a pinned SHA, or that grants write permissions to a job that only needs read. Each of these is syntactically valid. Each passes <code>actionlint</code>. None of them is visible as a problem until it's exploited or audited.</p>

<p>And under the current SBOM gap: none of them is traceable to "this was generated by a model, reviewed by nobody, and merged because the CI check passed."</p>

<h2 id="what-high-performers-do">What the high-performer pattern actually looks like</h2>

<p>geohot's high-performer/agent combination works because the high performer can evaluate the output. In security engineering that means: can you read a generated IAM policy and immediately identify whether it's too permissive? Can you read generated Terraform and spot the resource naming conflict before apply? Can you read a generated GitHub Actions workflow and verify the secret handling?</p>

<p>If yes, agents make you faster. If no, agents make you confidently wrong faster.</p>

<p>The practical controls that make the high-performer pattern work for security code:</p>

<p>Commit-level attribution. A policy requiring commits containing AI-generated security-relevant code to be labeled as such costs nothing to implement and creates the audit trail that incident response will ask for. Pre-commit hook or CI check; neither requires procurement.</p>

<p>Explicit security review gates. Branch protection rules that require a named security reviewer to approve commits flagged as AI-assisted before merge to protected branches. Not an automatic CI check — a human who can actually evaluate whether the security properties are correct.</p>

<p>Policy-as-code validation before merge. For infrastructure code: OPA/Rego gates that validate generated Terraform against the intended architecture, not just against syntax. A gate that checks "does this VPC peering connect networks that should not be connected" catches what <code>terraform validate</code> cannot.</p>

<p>These are not exotic controls. They're the same controls that apply to any other third-party input to a build process. The gap isn't that the controls are unavailable — it's that AI-generated code has been tacitly exempted from them because it looked like it came from a team member.</p>

<h2 id="the-compounding">The compounding that matters</h2>

<p>The Eternal Sloptember problem and the SBOM provenance problem are both manageable in isolation. A quality problem is caught by review. A provenance gap is addressed by attribution policy. Neither requires new tooling.</p>

<p>They compound because the quality problem is most severe in exactly the cases where the provenance gap is most likely: code generated quickly, by someone without deep context on the system, merged because it looked right, with no record that it was generated at all.</p>

<p>The incident that surfaces this will not look like "we deployed AI-generated code that was wrong." It will look like "we have a misconfiguration in our pipeline and we can't reconstruct how it got there." The blast radius will be real before the cause is visible.</p>

<p>Build the controls before the incident. The provenance trail is a morning's work. The policy gate is an afternoon's work. The review requirement is a settings change. None of this requires abandoning AI-assisted development — it requires treating AI-generated code like what it actually is: third-party input to your build process, with all the governance that implies.</p>
