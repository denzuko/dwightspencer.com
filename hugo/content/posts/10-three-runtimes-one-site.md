parentheses. The PostScript section assumes you have read
<a href="https://www-sop.inria.fr/members/Manuel.Serrano/publi/pot.pdf">PoC&#8741;GTFO</a>
at least once.</em></p>

<h2 id="the-premise">The premise</h2>

<p>Most personal sites are publishing targets. You write content, a tool
transforms it, and static files land on a server. The site has no
opinion about itself. It cannot be queried, reasoned over, or loaded
into a REPL. It just sits there.</p>

<p>This site is different in a specific, non-metaphorical way. It ships as
a Common Lisp system — an ASDF-loadable package with a micro-Prolog
knowledge base containing every post as a queryable fact, a generic
render multimethod with a PostScript backend, and a finger entry point
that produces the same output as the homepage. Three runtimes, one
codebase: Hugo builds the HTML, SBCL loads the system, Ghostscript
(or any PostScript interpreter) renders the corpus.</p>

<p>This post is about why that is useful and how it works — not as a
curiosity, but as an architectural pattern worth stealing.</p>

<h2 id="why-self-documenting">Why self-documenting systems</h2>

<p>The term comes from a specific tradition. Smalltalk images documented
themselves. Emacs Lisp environments document themselves. The UNIX
<code>man</code> pages were an early attempt, though they document programs rather
than the programs documenting themselves. The ideal — articulated at
Xerox PARC and refined at MIT — is a system that can be asked what it
is and answer correctly, from itself.</p>

<p>Static sites violate this. A Hugo site cannot tell you how many posts
it has from within the site. It cannot answer "which posts are tagged
infrastructure?" without a search index generated by a separate tool.
The content and the program that reasons about the content are
completely separate artifacts.</p>

<p>The alternative is to treat the site as a program that happens to also
emit HTML. The content becomes data in the program. Queries over the
content become function calls. The HTML output is one render target
among several.</p>

<h2 id="the-architecture">The architecture</h2>

<p>Three packages, three concerns:</p>

<pre><code>dsc/logic    — micro-Prolog engine (backward chaining, unification)
dsc/corpus   — post facts asserted into a knowledge base
dsc/render   — PostScript output via generic render multimethod
</code></pre>

<p>The corpus package is the connective tissue. Every post becomes a
fact assertion:</p>

<pre><code>(post   slug title date word-count)
(tag    slug tag-name)
(author slug orcid)
</code></pre>

<p>These are not stored in a database. They are asserted into a
<code>prolog-db</code> struct at load time — the same structure and interface as
the micro-Prolog engine in
<a href="https://github.com/denzuko/ml-prolog-pokemon">ml-prolog-pokemon</a>.
That project was an architecture study: could the same logic-engine
design that handles Pokémon move queries handle arbitrary domain facts?
Yes. The corpus here is posts; the logic engine does not care.</p>

<p>The corpus itself is generated by Hugo. Every <code>hugo --gc</code> produces a
fresh <code>corpus.lisp</code> from post front matter — a file that, when loaded,
asserts the current state of the site into the knowledge base. The
generator is a Hugo layout template. The output is valid Common Lisp.
The site generates part of its own program.</p>

<h2 id="querying-the-site">Querying the site</h2>

<p>Once loaded, the site is queryable as a program:</p>

<pre><code>&#35; posts tagged :infrastructure
(dsc/corpus:find-by-tag kb :infrastructure)

&#35; all posts, chronological
(dsc/corpus:all-posts kb)

&#35; unification query — all slugs with a given tag
(dsc/logic:db-prove-all kb '(tag ?slug :lisp))
</code></pre>

<p>The logic engine supports variables (prefixed <code>?</code>), unification, and
backward chaining. It is not a full Prolog — no cut, no assert/retract
at query time, no arithmetic. It does not need to be. The query surface
that matters for a corpus of posts is selection and projection, not
computation. The micro-Prolog is exactly sized to the problem.</p>

<h2 id="the-postscript-layer">The PostScript layer</h2>

<p>PostScript is not a render target anyone chooses in 2026. That is
precisely why it is interesting.</p>

<p>PostScript is Turing-complete. It has a stack, conditionals, loops,
and procedures. It was designed to describe pages, but it can describe
anything a stack machine can describe. The PoC&#8741;GTFO tradition has
demonstrated this extensively — PostScript files that are simultaneously
valid PDFs, valid ZIPs, valid ELF binaries.</p>

<p>The render layer here is more modest: a generic <code>render</code> multimethod
dispatched on output format. The PostScript backend emits a valid
<code>.ps</code> file describing the corpus — post titles, slugs, tags — in a
format readable by any PostScript interpreter and also readable as
Common Lisp comments. The file is simultaneously a PostScript program
and an annotated Lisp document.</p>

<pre><code>&#35; render the corpus to PostScript
(dsc/render:render :ps *standard-output* kb)
</code></pre>

<p>The PostScript backend exists to prove that the generic dispatch works
and to maintain the tradition that output formats are policy, not
mechanism.</p>

<h2 id="hugo-as-build-system">Hugo as build system</h2>

<p>Hugo is not the site. Hugo is the build system that produces one render
target — HTML — and generates part of the Lisp source (<code>corpus.lisp</code>).
This distinction matters.</p>

<p>The Hugo templates that generate <code>corpus.lisp</code> are the source of truth
for the machine-readable corpus. The front matter in each post is the
schema. The template is the ETL. The output is a Lisp program.</p>

<p>The same author identity data that drives the finger block on the
homepage — <code>hugo/data/author.yaml</code> — drives the <code>author</code> facts in
the corpus. One source, three representations: HTML, Lisp facts,
PostScript output.</p>

<h2 id="what-this-enables">What this enables</h2>

<p><strong>The site reasons about itself.</strong> A query over the corpus answers
questions that would require external tooling on a plain static site —
which posts share a tag, what the oldest post is, whether a slug
exists before generating a link to it.</p>

<p><strong>The site is loadable.</strong> An ASDF system with a Quicklisp dist means
the site can be pulled into any running SBCL image, queried, and
extended. The <code>dsc/logic</code> engine is the same code that runs in
<a href="https://github.com/denzuko/ml-prolog-pokemon">ml-prolog-pokemon</a> —
the corpus project and the game architecture study share an engine.</p>

<p><strong>The render target is swappable.</strong> The generic <code>render</code> multimethod
means adding a new output format — JSON-LD, Turtle, OPML — is a new
method, not a new pipeline. The mechanism is stable; the policy is
extensible.</p>

<p><strong>The system documents itself accurately.</strong> The finger block on the
homepage is generated from <code>hugo/data/author.yaml</code> and rendered as a
real SBCL-executable <code>defpackage</code> form. Running it produces the same
output as reading the page. The documentation is the program.</p>

<h2 id="the-pattern">The pattern, abstracted</h2>

<p>The specific technology choices — Common Lisp, Prolog, PostScript,
Hugo — are less important than the pattern:</p>

<ol>
<li><strong>Content is data.</strong> Treat every piece of content as a fact in a
knowledge base, not as a document in a file system.</li>
<li><strong>The build system generates part of the program.</strong> Hugo emitting
<code>corpus.lisp</code> is the same pattern as a code generator emitting type
definitions from a schema — the schema is authoritative, the
generated code follows.</li>
<li><strong>Output format is policy.</strong> Generic dispatch over output formats
means the mechanism is stable and the policy is extensible without
touching the core.</li>
<li><strong>The system answers questions about itself.</strong> A system that can be
queried about its own content is easier to maintain, extend, and
verify than one that cannot.</li>
</ol>

<p>This pattern is not new. It is the pattern of Smalltalk images, of
Emacs, of the original Lisp machines. It fell out of fashion when
the industry standardized on pipelines that transform content into
artifacts. It is worth recovering.</p>

<h2 id="implementation">Implementation</h2>

<p>Requires SBCL and Quicklisp. Source is under <code>hugo/static/lisp/</code>
in the <a href="https://github.com/denzuko/dwightspencer.com">site repository</a>.
The corpus template is <code>hugo/layouts/index.lisp</code>.</p>

<p>Once loaded, the corpus reflects the site at build time — <code>make-site-kb</code>
calls <code>assert-post-facts</code> on a fresh <code>prolog-db</code>, populating it from
the facts compiled into the system when Hugo generated <code>corpus.lisp</code>.
The knowledge base is current as of the last site build, which is
the correct guarantee: a static site's facts are facts at deploy time.</p>
